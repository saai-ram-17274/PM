## Legend

| | Meaning |
|---|---|
| ✅ | Ships today in Log360 Cloud — data exists in the documented backend column / ES index / API and is already surfaced somewhere in product |
| 🟡 | Partial — raw data exists but the **specific projection/join/UI** required by V5 is not pre-built. Solvable with correlation rule or thin enrichment layer. No new collector needed. |
| 🟠 | Source data not collected by default — needs a new log-source onboarding (e.g. Sysmon, EDR, DLP) or an additional collector enabled |
| ❌ | Not in product — requires net-new capability (e.g. AI prediction layer, sensitivity-label sync) |
| 🤖 | AI-only field (verdict text, recommendation, explanation, classification) — no backend dependency, generated by LLM |

**Aggregate verdict:** ~75% of slider fields ship today. ~15% are joins/aggregations on data we already ingest (partial). ~8% require turning on a log source the customer doesn't always send today (Sysmon/EDR-class telemetry). ~2% require net-new capability.

The single biggest gap pattern: **process-level forensic telemetry** (`proc-powershell`, registry mods on `svc-winupdatesvc`, AMSI events on `dev-ws045`) requires Sysmon / Defender for Endpoint / a real EDR. Log360 Cloud has the ingestion path but customers must onboard those sources.

## What V5 needs from Log360 Cloud to ship this prototype to a real customer

1. **Entity-graph correlation engine** that joins per-source events into the 24 canonical edges (most pieces exist piecewise; the join layer is the build). Reference: [`relation_catalog.md`](relation_catalog.md).
2. **Three lightweight enrichers**: (a) haversine "impossible speed" compute, (b) MFA-method classifier ("replay" / "fatigue" / "approved"), (c) OAuth-token-fingerprint linker (`correlationId` across SignInLog rows).
3. **Source onboarding checklist surfaced in the alert** — for every 🟠 above, tell the analyst "this slider section is empty because Sysmon/AMSI/Graph API logs aren't onboarded; here's the 1-click onboarding link."
4. **Sensitivity-label sync** (M365 → Log360 Cloud) so SharePoint slider can resolve `SensitivityLabelId` to display name.
5. **Threat-feed extension** to retain VirusTotal vendor counts (not just verdict) — small schema add on `ADSThreatAnalyticsFeeds`.
6. **AI verdict layer** — for every 🤖 above. This is the **net-new** capability V5 is pitching; everything else is wiring.

## Bottom line

The slider experience V5 prototypes is **mostly assemble-from-existing-parts** for the 12 entities. The two real engineering investments are (a) the **entity-graph correlation/join layer** that consolidates events into a navigable graph, and (b) the **AI verdict/recommendation layer**. Everything else is either already shipping in Log360 Cloud, a thin enrichment, or a customer-side log-source onboarding step.
